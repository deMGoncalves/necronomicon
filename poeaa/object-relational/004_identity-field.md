# Identity Field

**Classification**: Object-Relational Structural Pattern

---

## Intent and Purpose

Save database object ID as a field in the object to maintain identity between memory and database. Uses database primary key as object identity.

## Also Known As

- ID Field
- Primary Key Field
- Object ID

## Motivation

To persist an object in relational database and retrieve it later, you need a way to identify which row corresponds to which object. Identity Field solves this by storing the row's primary key as an object property. When object is saved, ID is generated or updated. When object is loaded, ID comes along allowing future updates.

Key decisions include: using numeric auto-incremented keys, GUIDs, or compound keys; if ID is immutable or can change; if generated by database or application. Each choice affects performance, portability, and complexity. Identity Field makes ID an explicit part of the object model.

## Applicability

Use Identity Field when:

- Objects need to be persisted in relational database
- Unique identity of objects needs to be maintained between sessions
- Identity Map is used for cache
- Relationships need to be mapped via foreign keys
- Updates and deletes need to identify specific row
- Any persistence pattern (Active Record, Data Mapper) is used

## Structure

```
Domain Object
└── id: Identity Field

Implementation Options:
1. Simple Key
   └── id: number (auto-increment)

2. GUID Key
   └── id: string (UUID/GUID)

3. Composite Key
   └── id: { orderId: number, itemId: number }

4. Meaningful Key
   └── id: string (email, username)

Database Table
└── PRIMARY KEY column
```

## Participants

- **Domain Object**: Object containing Identity Field
- [**Identity Field**](004_identity-field.md): Property storing primary key
- **Key Generator**: Generates new IDs (database or application)
- [**Data Mapper**](../data-source/004_data-mapper.md): Uses ID to correlate objects with rows

## Collaborations

- When new object is created, ID is null or temporary value
- When object is inserted, ID is generated (by database or application)
- Generated ID is stored in object's Identity Field
- When object is loaded, ID comes from database with data
- For update, Data Mapper uses ID to identify row
- For delete, ID identifies which row to remove
- Identity Map uses ID as key for cache

## Consequences

### Advantages

- **Simple identity**: Straightforward way to identify objects
- **Performance**: Numeric keys are fast for indexes
- **Uniqueness**: Guarantees unique identity in database
- **Relationships**: Facilitates foreign key mapping
- **Portability**: Universal concept in relational databases
- **Cache**: Allows efficient use of Identity Map

### Disadvantages

- **Exposes persistence details**: ID is technical detail in domain
- **ID generation**: Coordinate generation between app and database
- **Composite keys**: Additional complexity
- **ID change**: Difficult if ID needs to change
- **GUIDs**: Occupy more space than integers
- **Meaningful keys**: Can change, violating identity

## Implementation

### Considerations

1. **Key type**: Numeric auto-increment, GUID, or natural key
2. **Generation**: Database-generated vs application-generated
3. **Immutability**: ID should be immutable after creation
4. **Visibility**: ID part of public interface or internal
5. **Composite keys**: How to handle compound keys
6. **Sequence vs Identity**: PostgreSQL sequences vs MySQL AUTO_INCREMENT

### Techniques

- **Auto-increment**: Let database generate sequential IDs
- **GUID/UUID**: Generate GUIDs in application
- **Hi/Lo Algorithm**: Generate IDs in batches for performance
- **Sequence Object**: Use database sequences
- **Natural Key**: Use business attribute as ID (avoid)
- **Composite Key**: Use compound object as ID

## Known Uses

- **Hibernate @Id**: Id annotation in JPA/Hibernate
- **Entity Framework Key**: Primary key in EF
- **ActiveRecord id**: Default id field in Rails
- **Sequelize primaryKey**: Primary key in Sequelize

- **SQLAlchemy primary_key**: Column with primary_key=True
